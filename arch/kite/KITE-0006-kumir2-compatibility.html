1. Цель и принципы

Режим совместимости решает три задачи:

  • Запуск старых программ КуМир 2 без изменений (совместимость по поведению).
  • Переключатели поведения — спорные места (видимость, кодировка, деление) ведут себя «по-старому».
  • Исполнители — классические Робот, Чертёжник и др. доступны как часть совместимости.
Не «урезание» Режим не прячет возможности КуМир 3 — он меняет поведение на совместимое и включает исполнителей. Расширения 3 остаются доступны, если не конфликтуют с семантикой 2.
Статус Единого режима в движке ещё нет. Сейчас лексер понимает ключевые слова 2 и 3 одновременно, в kumir.toml есть поля edition/kumir_version, строки умеют CP-1251 — но переключателя нет. Всё ниже помечено 📋 Предложено.

2. Активация и приоритет

Три способа включения, с понятным приоритетом (сильнее — выше):

ПриоритетСпособОбластьДоступность
1 (сильнейший)Директива-комментарий в начале файлаТолько этот файл📋
2Флаг CLI --режим кум2Весь запуск📋

Директива файла

Директива оформлена как обычный комментарий Кумира (|) — поэтому классическая среда КуМир 2 просто игнорирует строку, и файл остаётся запускаемым «как есть».

| MODE: KUM2          | по-английски
| РЕЖИМ: КУМ2         | эквивалент по-русски

алг Главный
нач
  | ...классический код КуМир 2...
кон

Настройка в kumir.toml

edition задаётся числом; для точного закрепления версии языка — отдельное поле kumir_version (семантическая версия).

[package]
edition = 2                 # 2 = КуМир 2, 3 = КуМир 3
kumir_version = "2.1.0"     # точный pin диалекта/семантики

3. Гранулярность и смешивание

Режим действует на уровне файла/модуля. Это позволяет смешивать:

  • Проект в целом — на КуМир 3 (по умолчанию или через kumir.toml).
  • Отдельные файлы — в совместимости через директиву, не затрагивая остальные.
  • И наоборот: проект с edition = "кумир2" глобально, а отдельный новый файл — на КуМир 3.
Граница — модуль Режим фиксируется при загрузке файла и не «протекает» в импортированные модули — каждый модуль объявляет свой режим сам.

5. Исполнители

Классические «исполнители» — визитная карточка КуМир: учебные миры с состоянием (обстановкой) и набором команд. В режиме совместимости они доступны как стандартные модули.

ИсполнительСредаКоманды (примеры)Доступность
РоботКлетчатое поле со стенамивверх, вниз, влево, вправо, закрасить; проверки слева свободно📋
ЧертёжникКоординатная плоскостьопустить перо, поднять перо, сместиться в точку📋
ВодолейСосуды с водойнаполнить, перелить, вылить📋

Модель исполнителя Каждый исполнитель = состояние обстановки + набор команд + условия отказа (упёрся в стену и т.п.). Обстановка загружается из файла поля; команды меняют состояние и могут возбуждать отказ.

6. Как это работает внутри

Флаг режима протягивается через конвейер исполнения:

  1. Загрузчик определяет режим файла (директива → CLI → kumir.toml → КуМир 3).
  2. Лексер выбирает кодировку (CP-1251/UTF-8) и при необходимости ограничивает набор лексем.
  3. Парсер/семантика применяет правила видимости и объявления согласно режиму.
  4. Среда исполнения регистрирует исполнителей и совместимую стандартную среду.
Зависимость Полноценные переключатели видимости опираются на переработку окружения (KITE 4): сначала появится лексическая видимость в КуМир 3, затем — режим, возвращающий старое поведение.

7. Миграция

  • Запусти старую программу в режиме кумир2 — она работает как раньше.
  • Включай возможности КуМир 3 пофайлово, по мере готовности.
  • Линтер кумлинт подсказывает, какие места изменят поведение при переходе на КуМир 3.

8. Открытые вопросы (думаем вместе)

Уже решено
  • Синтаксис директивы: комментарий | MODE: KUM2 / | РЕЖИМ: КУМ2 — чтобы файл оставался запускаемым в классическом КуМир 2.
  • edition: числовое значение (2 или 3); точный pin — через kumir_version.
Ещё обсуждаем
  • Полный список переключаемых различий поведения (раздел 4) — что ещё отличается (деление, округление, имена встроенных величин)?
  • Канонический набор команд каждого исполнителя и формат файла обстановки.
  • Допустимо ли в режиме КУМ2 частично использовать фичи 3 — и какие именно запретить.
