1. Два вида библиотек

  • Встроенные — реализованы в ядре на Rust, дают нативную скорость и доступ к системе.
  • Пользовательские — написаны на Кумире (.kum), исполняются интерпретатором.
  • Нативные расширения — скомпилированные модули на C / C++ / Rust, подключаемые как библиотеки (см. раздел 9). 📋

Подключаются они единообразно; для пользователя разница невидима.

3. Пользовательские библиотеки

Самый простой вариант — один .kum файл с алгоритмами:

| моя_библиотека.kum
алг цел удвоить(цел x)
нач
  знач := x * 2
кон

И его использование:

использовать "моя_библиотека.kum"

алг Главный
нач
  вывод удвоить(5)   | 10
кон

Полноценная библиотека — это директория с конфигурацией:

моя_библиотека/
├── kumir.toml      | конфигурация
├── src/lib.kum     | исходный код
├── README.md       | документация
└── examples/       | примеры
Автоопределение типа Путь на .kum → простая библиотека; директория с kumir.toml → проект; имя без расширения → встроенная библиотека.

4. Конфигурация kumir.toml

[package]
name = "моя_библиотека"
version = "1.0.0"
authors = ["Имя <email@example.com>"]
description = "Описание"
license = "MIT"
type = "library"
main = "src/lib.kum"
exports = ["удвоить", "утроить"]

[dependencies]
время = "^2.0"

[build]
optimization_level = 2
strict_mode = true

[library]
category = "математика"
tags = ["math", "utils"]
stable = true

Из конфигурации извлекаются имя, версия, авторы, описание, зависимости, настройки сборки и список экспортируемых функций.

6. Как это работает внутри

  1. Менеджер библиотек (LibraryManager) регистрирует функции библиотек и хранит загруженные.
  2. Встроенные функции имеют нативный обработчик (Rust); пользовательские исполняются интерпретатором.
  3. Импортер файлов (FileImporter) загружает и парсит .kum-модули.
  4. Квалифицированный вызов Библиотека.функция(...) направляется менеджеру библиотек.
  5. Загруженные библиотеки кэшируются — повторный импорт берёт готовую версию.

7. Версии, зависимости, окружения

  • SemVer и спецификации версий (2.0.0, ^1.5) — для зависимостей и импорта.
  • Резолвер зависимостей строит граф и разрешает версии. 🧪
  • Виртуальные окружения изолируют наборы версий библиотек на проект. 🧪

8. Ограничения

  • Пользовательские библиотеки не имеют нативных обработчиков — функции идут через интерпретатор.
  • Рекурсивные (циклические) зависимости пока не поддерживаются.

9. Нативные расширения (C/C++/Rust)

По аналогии с Python (C-расширения), Rust (крейты) и другими языками, Kumir 3 предусматривает нативные расширения — скомпилированные модули на «верховных» языках цепочки (C, C++, Rust), которые экспортируют функции и типы в интерпретатор. Это путь к высокой производительности и переиспользованию существующих нативных библиотек.

Основа уже есть в ядре: нативные функции (NativeFn), которыми пользуются встроенные библиотеки, и тип @native для нативных объектов (см. KITE 2). Расширения обобщают этот механизм на внешние модули.

ЯзыкКак подключаетсяДоступность
RustКрейт с экспортом через мост Kumir (аналог pyo3); самый «родной» путь — ядро уже на Rust.📋
C / C++Динамическая библиотека со стабильным C-ABI; функции регистрируются в реестре нативных функций.📋
Принципы моста (дизайн)
  • Стабильный ABI и заголовок Kumir: преобразование Value ↔ нативные типы.
  • Манифест расширения в kumir.toml (type = "native", целевые платформы, символы).
  • Изоляция и безопасность: расширения помечаются как доверенные; песочница их ограничивает.
  • Кроссплатформенная сборка через кумпак: исходники → бинарь под ОС пользователя.
| Использование нативного расширения неотличимо от обычной библиотеки:

10. Предложенное

  • Пакетный менеджер кумпак и онлайн-реестр: установка и публикация библиотек. 📋
  • Импорт по URL из доверенных источников. 📋
  • Файл блокировки версий для воспроизводимых сборок. 📋
  • Опциональные зависимости и фичи-флаги — собирать только нужные части библиотеки. 📋
  • Рабочие пространства (workspaces) — несколько связанных пакетов в одном репозитории. 📋
  • Платформенные зависимости — разные модули под Windows/macOS/Linux. 📋
  • Пространства имён и ре-экспорт — аккуратная организация публичного API библиотеки. 📋
  • Подписи и контрольные суммы пакетов — безопасность цепочки поставок. 📋
  • Документация через кумдок — генерация страниц API из исходников и метаданных. 📋
  • Политика устаревания — [DEPRECATED]-пометки и предупреждения при сборке. 📋
