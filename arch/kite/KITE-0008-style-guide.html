1. Принципы

  • Ясность важнее краткости. Код читают чаще, чем пишут.
  • Code as Narrative. Программа должна читаться как связный рассказ.
  • Образование прежде всего. Не выпендривайся синтаксисом — объясняй намерение.
  • Меньше сюрпризов. Делай очевидное очевидным способом.

  • Логические величины называй утверждением: есть_решение, пусто_ли.

3. Структура программы

  • Один алгоритм — одна задача. Если алгоритм не помещается на экран — раздели его.
  • Используй контракты дано/надо для предусловий и постусловий.
  • Явно помечай аргументы: арг, рез, аргрез.

4. Комментарии

  • Комментарий | объясняет почему, а не пересказывает что.
  • Документируй назначение алгоритма строкой-комментарием перед нач.
  • Не оставляй закомментированный «мёртвый» код — для истории есть git.

5. Работа с типами

  • Указывай типы на границах (параметры, возврат); внутри допустимо авто.
  • Для «может не быть значения» используй T? вместо пусто/null.
  • Для операций, которые могут не удаться, — T!E вместо исключений (см. KITE 2).
  • Не приводи типы молча — сужающие преобразования делай явно.

6. Управление потоком

  • Длинные если-иначе-если заменяй на совпадение (match) с охранами когда.
  • Выходи рано: проверяй ошибочные случаи в начале, а не вкладывай счастливый путь глубоко.
  • Цепочки преобразований оформляй конвейером |>.

7. Ошибки и инварианты

  • Возвращай T!E и пробрасывай ошибку оператором ?.
  • Инварианты, которые «не должны нарушаться», проверяй через утв (assert).
  • Сообщения об ошибках пиши понятно — их прочитает ученик.

8. Переменные и видимость

  • Объявляй переменные перед использованием — не полагайся на неявное создание (см. KITE 4).
  • Не опирайся на стековую (динамическую) видимость — передавай данные через параметры.
  • Помечай неизменяемое как конст; измен — только когда мутация действительно нужна.
  • Область переменной — минимально необходимая.

9. Форматирование

  • Отступ — кратный (например, 2 пробела); не смешивай табы и пробелы.
  • Один оператор — одна строка; пробелы вокруг бинарных операторов.
  • Держи строки умеренной длины (ориентир ~100 символов).
  • В ядре (Rust) используй секции-баннеры и статус-теги BetterDoc (см. BDoc).
  • Доверяй форматтеру кумфмт, когда он появится — единый стиль без споров.
