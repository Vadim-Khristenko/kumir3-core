1. Текущее состояние

Честно AST-модель ООП богатая (классы, структуры, интерфейсы, типажи, impl, generics, модификаторы виртуальный/переопределить/финал/абстрактный, видимость). Уже сделано: идентичность объектов (стабильный type_id через TypeRegistry) и разрешение методов по иерархии с работающим переопределением (полиморфизм). Осталось: полные проверки виртуальности, диспетчеризация типажей/интерфейсов, инкапсуляция, generics, перегрузка операторов.

Ниже: ✅ — работает, 🧪 — частично/в AST, 📋 — спроектировано.

2. Модель типов

ВидНазначениеДоступность
классКласс с наследованием, методами, виртуальной диспетчеризацией.🧪
структураЛёгкие данные без виртуальной диспетчеризации.🧪
интерфейсТолько сигнатуры методов.🧪
свойство (trait)Типаж с дефолтными реализациями.🧪

Член типа описывается полем (Field), методом (Method), конструктором/деструктором; у каждого — видимость и модификаторы.

3. Наследование и полиморфизм

  • Наследование: расширяет (один предок), реализует (интерфейсы).
  • Разрешение методов по иерархии: поиск метода идёт от класса объекта вверх по предкам — первый найденный выигрывает (vtable-подобно). ✅
  • Виртуальная диспетчеризация: вызов по фактическому типу (переопределение выигрывает); финал запрещает переопределение, абстрактный требует реализации в неабстрактном потомке. ✅
  • Вызов предка: предок.метод(...) — вызов реализации родителя (обходит переопределение). ✅
  • Абстрактные классы: создание экземпляра абстрактного класса запрещено. ✅
Фундамент — идентичность объектов Корректная диспетчеризация требует надёжной связи объект → класс (через TypeId и TypeRegistry), а не сопоставления по набору полей. Это первый шаг плана (§9).

4. Интерфейсы и типажи

  • Интерфейс — набор сигнатур; тип обязан их реализовать.
  • Типаж (свойство) — как интерфейс, но с дефолтными реализациями методов и супертипажами.
  • Реализация: реализация Типаж для Тип привязывает методы к типу.
  • Диспетчеризация через impl-блоки: метод из реализация [Типаж] для Тип ищется по типу и предкам после методов класса. ✅
  • Дефолтные реализации типажа и проверка соответствия интерфейсу — на очереди. 📋
  • Проверка реализации через TypeRegistry (номинальный подтип, см. KITE 10). 🧪

6. Обобщённые типы

  • Параметры типа (type_params) на классах, методах, интерфейсах и типажах.
  • Ограничения где — требования к параметрам (реализует типаж, подтип и т.п.).
  • Представление через TypeKind::Generic { name, type_args }; проверка совместимости — движком (ковариантно по аргументам).
  • Стратегия: стирание с проверкой типов на границах (мономорфизация — позже). 📋

7. Перегрузка операторов

  • Операторы выражаются типажами-операторами (напр. Сложение, Сравнение); a + b разворачивается в вызов метода типажа.
  • Тип результата операции даёт движок (result_of_binop) либо сигнатура метода типажа.
  • Это позволяет дробь, комплекс, вектор и т.п. вести себя как числа. 📋

8. Связь с движком типов

ООП опирается на единый движок (KITE 10):

  • Номинальный подтип (nominal-правило) — наследование и реализация интерфейсов через TypeRegistry.
  • Проверка присваиваний и параметров методов — check_assignable.
  • Типы полей и значения по умолчанию — default_value.
  • Перегрузка операторов — типизация через result_of_binop.

9. План реализации

  1. ✅ Идентичность объектов: класс регистрируется в TypeRegistry при объявлении; объект несёт стабильный type_id (вместо сопоставления по полям).
  2. ✅ Разрешение методов по иерархии: поиск метода от класса объекта вверх по предкам; переопределение подкласса выигрывает (полиморфизм).
  3. ✅ Виртуальная диспетчеризация: переопределение, предок.метод(), запрет инстанцирования абстрактный, запрет переопределения финал, обязательность реализации абстрактных методов.
  4. 🚧 Типажи и интерфейсы: диспетчеризация в impl-методы — ✅; дефолтные реализации типажа и проверка соответствия интерфейсу — далее.
  5. 🚧 Инкапсуляция и свойства: проверка видимости полей — ✅; свойства-аксессоры и типизация полей через движок — далее.
  6. Обобщённые типы: параметры и ограничения где.
  7. Перегрузка операторов: типажи-операторы.
