1. Модель окружения

Среда выполнения (Environment) состоит из:

  • Глобальная область (globals) — переменные и константы уровня программы.
  • Стек кадров вызова (call_stack) — по кадру (CallFrame) на каждый активный вызов алгоритма.
  • Реестры алгоритмов, классов, интерфейсов, типажей, реализаций, перечислений и нативных функций.

Каждый кадр вызова хранит: имя алгоритма, локальную область (locals), возвращаемое значение знач и текущий объект я/это (для методов).

2. Область видимости (Scope)

Область видимости — это две таблицы: изменяемые переменные и неизменяемые константы. При чтении сначала проверяются переменные, затем константы.

Гранулярность — на уровне вызова Отдельные блоки (если, нц…кц) не создают собственных областей видимости. Область одна на весь кадр вызова алгоритма (плюс глобальная). ✅ Текущее поведение

3. Рекурсивный поиск переменной

Поиск рекурсивен по цепочке лексических областей текущего кадра (от внутренней к внешней), затем — в глобальной области. Кадры вызывающих не просматриваются:

прочитать(имя):
    для области в текущем_кадре от внутренней к внешней:
        если в области есть имя:  вернуть значение
    если в globals есть имя:      вернуть значение
    иначе:                        ошибка «переменная не определена»
Лексическая видимость ✅ Реализовано Алгоритм видит только свои области (параметры, блочные) и глобальные — но не локали вызывающего. Прежняя стековая (динамическая) утечка устранена. Старое поведение доступно через режим совместимости КуМир 2 (KITE 6).

4. Присваивание и неявное создание

Запись имя := значение работает так:

  1. В текущем кадре (от внутренней области к внешней): если имя уже есть — записать туда (ошибка, если это константа).
  2. Иначе, если имя есть в глобальной области — записать туда (ошибка, если константа).
  3. Иначе — создать новую локальную переменную во внутренней области текущего кадра.
Неявное создание Присваивание необъявленному имени не ошибка — оно молча создаёт переменную. Удобно, но опечатка в имени порождает новую переменную вместо изменения нужной. ✅ Текущее поведение

5. Константы

Константы (конст) хранятся отдельно от переменных. Попытка присвоить значение константе — ошибка времени выполнения: «Нельзя изменить константу …». Защита действует в той области, где константа объявлена.

6. Рекурсия и защита глубины

Каждый вызов добавляет кадр в стек; выход — снимает. Глубина ограничена (max_call_depth, по умолчанию 1000). При превышении — ошибка:

Превышена максимальная глубина вызова (1000).
Возможно, бесконечная рекурсия.

Это защищает от переполнения стека при невыходящей рекурсии и заменяет «тихий» крах понятным сообщением.

7. Непредвиденные поведения

Бо́льшая часть прежних сюрпризов устранена лексической видимостью. Состояние:

СюрпризСостояние
Утечка видимости (алгоритм видел локали вызывающего)✅ Исправлено — поиск только в своём кадре.
Запись «не туда» (присваивание в кадр вызывающего)✅ Исправлено
Кросс-кадровое затенение✅ Исправлено
Опечатка → новая переменная⚠ Осталось — присваивание всё ещё неявно создаёт переменную (явное объявление пока не обязательно).
Нет блочной видимости в нц/если🧪 Частично — блочные области уже работают в совпадение/перехват; для нц/если — позже.

Тот же пример теперь ведёт себя корректно — помощник не видит чужую переменную:

алг помощник
нач
  вывод x        | ОШИБКА: переменная x не определена (лексическая видимость)
кон

алг Главный
нач
  цел x
  x := 42
  помощник       | x не виден внутри помощник — передавай параметром
кон

8. Направление

  • Лексическая видимость ✅ Реализовано — алгоритм видит только свои области и глобальные, не локали вызывающего.
  • Блочные области 🧪 Частично — есть инфраструктура (push_scope/pop_scope), задействована в совпадение/перехват; для нц/если — на очереди.
  • Обязательное объявление 📋 Предложено — присваивание необъявленному имени станет предупреждением/ошибкой (опечатки ловятся сразу).
  • Диагностика линтера 📋 Предложено — кумлинт о затенении и неиспользуемых переменных.
Совместимость Переход к лексической видимости — это изменение поведения; оно будет вводиться аккуратно (вероятно, под флаг режима совместимости — см. будущий KITE про КуМир 2).

9. Дополнительные улучшения

Идеи, делающие работу с окружениями мощнее, безопаснее и удобнее (все 📋 Предложено):

УлучшениеЧто даёт
Лексические замыканияЛямбды/вложенные алгоритмы захватывают переменные по месту объявления — предсказуемо и без утечек по стеку.
Явные динамические переменныеЕсли стековая видимость нужна намеренно — пометка динам/контекст; «случайная» утечка из раздела 7 становится осознанной фичей.
Блочные области { … }Переменные ограничены блоком если/нц; явные правила теневого объявления.
Параметры по умолчанию и именованные аргументыГибкая привязка значений к параметрам при вызове.
Task-local окруженияИзолированные переменные на асинхронную задачу — связь с KITE 3; без гонок.
Заморозка области (заморозить)Запрет на добавление/изменение имён в области — защита инвариантов.
Песочница окруженияИзолированное окружение с ограниченным набором имён — безопасный запуск недоверенного кода (онлайн-песочница).
Инспекция и watchAPI обхода окружения для отладчика/REPL: дамп переменных, наблюдение за изменениями.
Горячая перезагрузка в REPLПереопределение алгоритмов/переменных без перезапуска сессии.
Константы времени компиляцииВычисление конст-выражений до запуска; ранние ошибки.
Совместимость Стековое (динамическое) поведение остаётся доступным через режим совместимости — см. KITE 6.
