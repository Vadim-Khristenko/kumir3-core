1. Мотивация

Раньше отношения типов (совместимость, общий тип) были «зашиты» в большие match прямо на TypeKind и не были связаны с реестром номинальных типов. Добавить новый тип или правило означало править закрытые функции. Новый движок (shared/src/typesys.rs) делает систему расширяемой через регистрацию правил и единой для интерпретатора и компилятора.

2. Три слоя

СлойЧто описываетГде
Структурный — TypeKindФорма типа: числа, коллекции, обёртки, функции (см. KITE 2).types/value.rs
Номинальный — TypeRegistryИменованные типы, поля, методы, наследование.types/registry.rs
Движок — TypeSystemОтношения между типами: совместимость, унификация, операции.typesys.rs

Движок чист (без зависимостей от интерпретатора) и связывает первые два слоя.

Код разбит для чистоты: typesys/mod.rs — движок и базовые типы (Conformance, TypeError, TypeOp, Coercion), typesys/rules.rs — правила, typesys/tests.rs — тесты.

3. Уровни совместимости

Запрос «подходит ли источник для цели» возвращает один из уровней (Conformance):

УровеньСмыслНеявно?
ExactИдентичные типы.да
WideningБезопасное расширение (цел_16→цел_64, T→T?).да
CoercibleНужно явное/контролируемое приведение (сужение, любой→цел).нет
IncompatibleНет отношения.—

4. Правила (TypeRule)

Каждое отношение — это правило с двумя необязательными методами: совместимость и унификация. Правила опрашиваются по порядку; первое решительное мнение побеждает.

pub trait TypeRule: Send + Sync {
    fn name(&self) -> &'static str;
    fn assignable(&self, ts: &TypeSystem, target: &TypeKind, source: &TypeKind)
        -> Option<Conformance> { None }
    fn unify(&self, ts: &TypeSystem, a: &TypeKind, b: &TypeKind)
        -> Option<TypeKind> { None }
}

5. Встроенные правила

ПравилоЧто покрывает
identityОдинаковые типы → Exact.
top-bottomлюбой/авто (вершина) и никогда (низ).
null-optionпусто→T? и авто-обёртка T→T?.
numericЧисловая решётка: расширение/сужение по размеру и знаку.
collectionСтруктурная рекурсия (ковариантно): массивы, множества, словари, кортежи, Option/Result, ссылки, generics.
nominalНаследование через TypeRegistry (Кот→Животное).

6. Унификация и общий тип

Для ветвлений и бинарных операций движок ищет «наименьший общий супертип» (unify / common_type): числа → больший по размеру/точности, коллекции — рекурсивно, пусто с T → T?.

ts.unify(&Int16, &Int64)        // Some(Int64)
ts.unify(&Int32, &Float64)      // Some(Float64)
ts.unify(&Array(Int16), &Array(Int64)) // Some(Array(Int64))

7. Типизация операций

result_of_binop(op, left, right) даёт тип результата:

  • Арифметика двух чисел → их общий тип; + со строкой → строка (конкатенация).
  • Сравнения (=, <, …) совместимых типов → лог.
  • Логические (и, или) над лог → лог; иначе — ошибка типа.

8. Расширяемость

Новый тип или отношение добавляется регистрацией правила, без правки закрытых функций:

let mut ts = TypeSystem::new();
ts.register_rule(Arc::new(МоёПравило));   // и оно сразу участвует в выводе

Это и есть «умный детектор» вместо одного гигантского перечисления, о котором говорил KITE 2. Правило можно добавить и в начало (register_rule_front) — оно получит приоритет.

Делегированные задачи

Движку переданы и смежные задачи, чтобы интерпретатор и компилятор не дублировали логику:

МетодЧто делает
coercion(target, source)План приведения: Identity / Widen / Wrap (в T?) / Cast / Forbidden — что именно вставить в коде/рантайме.
default_value(type)Естественное значение по умолчанию типа (0 / пусто / Ничего) для инициализации переменных.
is_subtype(sub, sup)Единая проверка подтипа (структурная + номинальная).
result_of_binop(op, a, b)Тип результата операции.

9. Использование в интерпретаторе и компиляторе

  • Интерпретатор: проверка присваиваний, выбор перегрузки, типизация выражений во время выполнения.
  • Компилятор: статическая проверка типов и решения кодогенерации (общий тип ветвей, результат операции).

Поскольку движок живёт в shared и не зависит от рантайма, оба крейта используют один и тот же источник правды.

Единый источник Есть общий процессный движок typesys::default_engine() (стандартные правила). Легаси-помощники TypeKind::is_assignable_from и common_type теперь — тонкие фасады над ним: дублирующей логики типов больше нет, движок единственный.

10. План интеграции

  1. ✅ Движок и набор правил с тестами в модуле shared/src/typesys/.
  2. ✅ Делегированы движку: приведение (coercion), значения по умолчанию, проверка подтипа.
  3. Перевести проверки присваивания/операций интерпретатора на TypeSystem.
  4. Добавить статический проход проверки типов в компиляторе.
  5. Связать TypeKind ↔ TypeRegistry для номинальных типов из пользовательского кода.
  6. ✅ Старые is_assignable_from/common_type делегируют движку — дубля логики нет.
  7. Добавить вывод типов (авто) и обобщённые ограничения (где).
Принцип Заменяем движок поэтапно за стабильным фасадом — текущий функционал сохраняется, а система типов становится мощнее и расширяемее.
